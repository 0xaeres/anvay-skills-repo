# anvay-skill

## Use This Skill When
Use this skill for product orientation, grounded development, review, debugging, and deciding when
to query the Anvay product knowledge base. It provides the canonical mental model, domain language,
system map, data model, contracts, invariants, security rules, and known traps for the Anvay
platform itself.

## Product Snapshot
Anvay is a per-product RAG and curation pipeline that indexes code and documentation, enriches it
with LLM-generated facts, and serves it through a council of specialized agents with a hard
human-in-the-loop approval gate. [file: github:0xaeres/anvay-core/CONTRIBUTING.md:18] It is used by
developers and product teams to maintain living, evidence-backed product knowledge. The primary
runtime is a Python monorepo (`anvay-core`) with a FastAPI application, an MCP server, and a
LangGraph-based council orchestration layer, deployed as a containerized service with SQLite,
Qdrant, and FalkorDB as its core data stores. [file: github:0xaeres/anvay-core/CONTRIBUTING.md:18]
[file: github:0xaeres/anvay-core/docs/product-system-intelligence-architecture.md:61]

## Product Language
- **Source**: A configured origin of resources (e.g., a GitHub repository, a documentation site)
  that Anvay ingests. [file: github:0xaeres/anvay-core/CONTRIBUTING.md:18]
- **Resource**: A single addressable unit within a source, identified by a canonical URI (e.g., a
  file path, a page URL). [file: github:0xaeres/anvay-core/CONTRIBUTING.md:18]
- **Ingest**: The continuous process of diffing a source's resources, chunking changed resources,
  embedding them, and upserting into Qdrant. [file: github:0xaeres/anvay-core/CONTRIBUTING.md:18]
- **Chunk**: A semantically bounded segment of a resource (e.g., a function, a heading section)
  produced by tree-sitter for code or heading-based splitting for docs. [file:
  github:0xaeres/anvay-core/CONTRIBUTING.md:18]
- **Council**: The LangGraph-based multi-agent system that drafts, reviews, and approves product
  skills. It includes specialized agents like `architect`, `domain_expert`, and `quality_expert`.
  [file: github:0xaeres/anvay-core/CONTRIBUTING.md:283]
- **Skill**: A curated, version-controlled Markdown document that captures a specific aspect of a
  product (e.g., overview, architecture, a feature guide). Skills are the primary output of the
  council. [file: github:0xaeres/anvay-core/anvay/council/skill_catalog.py:55]
- **Proposal**: A draft skill generated by the council that must pass a human-in-the-loop approval
  gate before being committed to the product's skill repository. [file:
  github:0xaeres/anvay-core/CONTRIBUTING.md:18]
- **Graph Store**: A FalkorDB-derived product-system topology index that models services, modules,
  APIs, schemas, tests, and their relationships for multi-hop queries. [file:
  github:0xaeres/anvay-core/docs/product-system-intelligence-architecture.md:61]

## Capabilities And Workflows
1. **Continuous Ingestion** - Anvay monitors configured sources, diffs resources by content hash,
   chunks only changed resources on semantic boundaries, and upserts dense and sparse embeddings
   into Qdrant. [file: github:0xaeres/anvay-core/CONTRIBUTING.md:18]
2. **Graph-Based Topology Extraction** - During ingest, Anvay deterministically extracts a
   product-system topology graph (services, modules, APIs, dependencies) and stores it in FalkorDB
   for multi-hop reasoning. [file:
   github:0xaeres/anvay-core/docs/product-system-intelligence-architecture.md:61]
3. **Council-Driven Skill Generation** - A LangGraph council of specialized agents retrieves fresh
   evidence, drafts product skills, and enforces a structured template covering architecture, data
   models, contracts, and invariants. [file: github:0xaeres/anvay-core/CONTRIBUTING.md:283]
4. **Human-in-the-Loop Approval** - Every council-generated skill proposal must be explicitly
   approved by a human before it is committed and pushed to the product's skill repository. [file:
   github:0xaeres/anvay-core/CONTRIBUTING.md:18]
5. **MCP Server for Tool Access** - Anvay exposes its retrieval and skill management capabilities as
   tools via a stdio MCP server, enabling integration with AI assistants. [file:
   github:0xaeres/anvay-core/anvay/mcp_server/server.py]

## Data Model
- **Source**: Configured origin of resources. Fields: type (e.g., github, docs), connection details,
  sync schedule. Stored in SQLite. Invariant: a source must have a unique identifier and valid
  credentials. [file: github:0xaeres/anvay-core/CONTRIBUTING.md:18]
- **Resource**: A single addressable unit within a source. Fields: canonical URI, content hash
  (SHA-256), embedding config version, status (added/updated/removed/unchanged). Stored in SQLite as
  the sync source of truth. Invariant: a resource's URI must be unique within its source. [file:
  github:0xaeres/anvay-core/CONTRIBUTING.md:18]
- **Chunk**: A semantically bounded segment of a resource. Fields: text content, dense embedding,
  BM25 sparse vector, source resource URI, chunk index. Stored in Qdrant. Invariant: every chunk
  must be traceable back to its parent resource via URI. [file:
  github:0xaeres/anvay-core/CONTRIBUTING.md:18]
- **GraphNode**: A node in the FalkorDB product-system topology. Fields: type (service, module, API,
  schema, test), name, properties. Stored in FalkorDB. Invariant: nodes must be connected by at
  least one typed relationship (e.g., CALLS, IMPLEMENTS, DEPENDS_ON). [file:
  github:0xaeres/anvay-core/docs/product-system-intelligence-architecture.md:61]
- **Skill**: A curated Markdown document. Fields: title, tier, content sections (architecture, data
  model, contracts, etc.), evidence citations. Stored as a file in the product's skill repository.
  Invariant: every factual claim in a skill must be supported by at least one citation to ingested
  evidence. [file: github:0xaeres/anvay-core/anvay/council/skill_catalog.py:55]

## Interfaces And Contracts
- `POST /sources` - Add a new source to the ingestion pipeline. Contract: request body must include
  source type and connection details; returns source ID. [file:
  github:0xaeres/anvay-core/anvay/api/routes/sources.py]
- `GET /sources/{source_id}/resources` - List resources for a source with their sync status.
  Contract: returns paginated list of resources with URIs and statuses. [file:
  github:0xaeres/anvay-core/anvay/api/routes/sources.py]
- `POST /skills/approve` - Approve a drafted skill proposal. Contract: request body includes
  proposal ID; triggers commit and push to the skill repository. [file:
  github:0xaeres/anvay-core/CONTRIBUTING.md:2]
- `find_skills` (MCP tool) - Retrieve curated product skills by topic or tier. Contract: accepts
  query string; returns list of matching skill summaries. [file:
  github:0xaeres/anvay-core/CONTRIBUTING.md:2]
- `query_code_context` (MCP tool) - Look up code symbols, definitions, and references. Contract:
  accepts symbol name or file path; returns relevant code excerpts. [file:
  github:0xaeres/anvay-core/CONTRIBUTING.md:2]
- `hybrid_search_corpus` (MCP tool) - Perform open-ended semantic + keyword search across the entire
  ingested corpus. Contract: accepts natural language query; returns ranked list of chunks with
  citations. [file: github:0xaeres/anvay-core/CONTRIBUTING.md:2]
## Invariants And Constraints
1. Every factual claim in a skill must be supported by at least one citation to an ingested evidence
   chunk. [file: github:0xaeres/anvay-core/CONTRIBUTING.md:18]
2. A source manifest diff must use canonical resource URIs and SHA-256 content hashes to determine
   added, updated, removed, and unchanged resources; unchanged resources must be skipped during
   ingest.
3. SQLite is the sync source of truth for resource manifests; FalkorDB is a derived graph index and
   must not be used as the primary store for resource state.
4. The council must not commit a skill proposal without explicit human approval; the approval gate
   is hard and cannot be bypassed by automation.
5. Graph extraction during ingest is deterministic; LLM-based graph enrichment is only allowed when
   `ingestion.graph.mode="bounded_llm"` and the light model is reachable, and even then only for
   allowlisted, strict-JSON facts.

## How To Use The Knowledge Base
This skill is a map, not the territory. Trust it for orientation — product purpose, the system map,
contracts, invariants, and where things live. For anything you are about to change, read, or assert
as current, treat the knowledge base as source of truth and query it: code moves faster than this
skill.

Query the KB over MCP. Pick the tool by question shape:

- **Orientation / "which skill applies"** — `find_skills` then `get_skill`.
Start here to load product context before deeper retrieval.
- **General "how does X work" / open-ended evidence** — `evidence_search_corpus`.
Multi-channel retrieval (hybrid + grep + repo-map + graph-local + summaries), coverage-assessed. The
default for most questions.
- **Relational / "what calls what", "what depends on this", impact** —
`ask_product_graph`. Walks the per-product knowledge graph and synthesizes a cited answer. Use when
the question is about connections, not a single file.
- **Exact symbol / constant / string lookup** — `grep_corpus`. Deterministic
match against indexed chunks when you already know the token.
- **Symbol definition / signature lookup** — `query_code_context`.
- **Semantic / paraphrased search of code + docs** — `hybrid_search_corpus`
(dense + BM25 + rerank) when you want the low-level retrieval directly.

Rules of thumb:
- Cite file:line from retrieved evidence, not from this skill, when you make a
concrete claim about current code.
- If this skill and the KB disagree, the KB wins and this skill is stale —
report it via `report_outcome` so it gets re-validated.
- Prefer `ask_product_graph` for "why/how are these connected" and
`evidence_search_corpus` for "show me the relevant code".
## How To Work In This Product
1. Clone the `0xaeres/anvay-core` repository and install dependencies with `pip install -e
   ".[dev]"`. [file: github:0xaeres/anvay-core/CONTRIBUTING.md:2]
2. Run the test suite with `pytest` to validate your environment. Key test files include
   `tests/test_product_skill_validation.py` and `tests/test_drafter_faithfulness.py`.
3. Start the FastAPI application locally with `uvicorn anvay.api.app:app --reload` for API
   development.
4. Run the MCP server in development mode with `python -m anvay.mcp_server.server` for tool
   integration testing.
5. Follow the contribution guide in `CONTRIBUTING.md` for PR conventions: all changes must pass the
   existing test suite, and new features should include tests that cover the modified council agents
   or ingestion pipeline.
6. When adding a new tool to the MCP server, register it in `anvay/mcp_server/tools.py` and document
   its contract.
7. When adding a new chunker language, extend the tree-sitter configuration in
   `anvay/ingest/chunker.py`.

## Security And Secrets
- Source credentials (e.g., GitHub tokens, API keys) must be stored in the SQLite source
  configuration and never logged or exposed in API responses. [file:
  github:0xaeres/anvay-core/CONTRIBUTING.md:2]
- The MCP server runs over stdio and does not expose network ports; it inherits the host process's
  environment for secrets. [file: github:0xaeres/anvay-core/anvay/mcp_server/server.py]
- LLM API keys used for enrichment and council agents must be provided via environment variables and
  never hardcoded in configuration files.

## Known Traps
- **Trap**: Assuming the graph store (FalkorDB) is the source of truth for resource state. It is a
  derived index; SQLite holds the canonical manifest. Querying FalkorDB for resource existence or
  status will yield stale or incomplete data. [file: github:0xaeres/anvay-core/CONTRIBUTING.md:18]
- **Trap**: Bypassing the human-in-the-loop approval gate in test or development environments. The
  approval step in `anvay/skills/approval.py` calls `commit_and_push`, which will push to the
  configured repository even in dev mode if not properly mocked. [file:
  github:0xaeres/anvay-core/anvay/skills/approval.py]
- **Trap**: Modifying the required skill template sections (e.g., "Data Model", "Interfaces And
  Contracts", "Invariants And Constraints") without updating the council's parser in
  `anvay/council/agents/skill.py`. The parser expects exact heading names and will fail or skip
  content if they are renamed.
- **Trap**: Adding LLM-generated graph facts without setting `ingestion.graph.mode="bounded_llm"`
  and ensuring the light model is reachable. The default mode is deterministic; LLM enrichment will
  be silently skipped, leading to missing graph relationships.
